# 3.5.2 CSR/CSC Specific Access Patterns

## Design

- The backend treats CSR and CSC as different traversal layouts, not merely aliases with renamed fields.
- This affects:
  sparse iterator headers
  random-access helpers
  sparse output assembly
  reference kernels
  signature/runtime assumptions.

## Implementation

- Sparse iterator configuration in scheduling chooses:
  CSR -> `row_ptr`, `col_idx`
  CSC -> `col_ptr`, `row_idx`.
- `IRExprEmitter::visit(const IRTensorAccess&)` emits:
  sparse value loads as `tensor->vals[pointerVar]`
  random-access sparse reads as `sp_csr_get(...)` or `sp_csc_get(...)`
  dense array indexing for 1D and 2D dense tensors.
- `emitSparseOutputHelpers()` emits `sp_csr_get` and/or `sp_csc_get` only when needed by active inputs.

## Runtime Layout

- The generated `SparseMatrix` struct stores both CSR and CSC field names:
  `row_ptr`, `col_idx`
  `col_ptr`, `row_idx`
  `vals`.
- Conversion helpers alias fields so one struct can represent either format:
  CSR conversion sets `col_ptr = row_ptr`, `row_idx = col_idx`
  CSC conversion sets `row_ptr = col_ptr`, `col_idx = row_idx`.

## Sparse Output Cases

- Union/intersection assembly differs by whether rows or columns are the outer stored axis.
- Dynamic-row accumulation for sparse output also branches on primary format:
  CSR-oriented outer row accumulation
  CSC-oriented outer column accumulation.
- Reference kernels in `reference_emitter.cpp` mirror this distinction and use format-specific pointer/index arrays.

## Practical Impact

- CSR and CSC versions of the same algebraic expression can lower to different scheduled loop shapes and different emitted helper code.
- This is visible in tests and golden artifacts for baseline, interchange, blocking, and full schedules.
