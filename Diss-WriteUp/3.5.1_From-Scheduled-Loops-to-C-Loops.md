# 3.5.1 From Scheduled Loops to C Loops

## Design

- The scheduled loop tree is meant to be directly renderable to C control flow.
- Each scheduled header kind corresponds to a specific emitted loop skeleton.
- Loop-local statements are carried as structured `ir::IRStmt` nodes so codegen does not need to reconstruct side effects from raw strings in the common path.

## Implementation

- `CodeGenerator::emitScheduledLoop(...)` is the core renderer.
- It walks the scheduled tree recursively and handles four header classes.

## Header Mapping

- `DenseFor`:
  emits `for (int idx = lower; idx < upper; idx++)`.
- `SparseIterator`:
  emits pointer-driven loops such as:
  `for (int pA = A->row_ptr[i]; pA < A->row_ptr[i + 1]; pA++)`.
- `SparseMerge`:
  emits merge `while` loops and per-term pointer/end variables for union or intersection traversal.
- `Block`:
  emits a wrapper loop over block IDs, then `start` and `end` variables that bound the inner dense loop.

## Body Emission

- `preStmts` are emitted before children.
- `postStmts` are emitted after children.
- This is how scalar accumulators, workspace setup, final stores, and cleanup logic are placed structurally inside the scheduled loop nest.
- `emitScheduledComputeBody(...)` also emits compute-level `prologueStmts` and `epilogueStmts` around the root loop.

## Structured Expression/Statement Emission

- `IRExprEmitter` lowers:
  tensor accesses
  constants
  binary ops
  scalar variables
  selected intrinsic-like calls such as `relu` and `sigmoid`
  indexed accesses
  comparisons.
- `emitIRStmt(...)` handles declarations, assignments, frees, conditionals, loops, calls, and raw fallbacks.

## Design Consequence

- Scheduling owns traversal semantics.
- Codegen mostly renders what the scheduler already decided, which is the intended architecture.
