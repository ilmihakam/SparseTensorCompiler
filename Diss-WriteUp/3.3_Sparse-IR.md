# 3.3 Sparse IR

## Design

- The implementation uses a layered IR instead of one monolithic representation.
- The split exists because sparse compilation needs both semantic facts and executable traversal structure.
- Shared `ir` utilities hold reusable structured expression and statement nodes.
- Semantic IR describes tensor computations before concrete loop scheduling.
- Scheduled IR converts that description into loop headers, sparse iterators, merge logic, and loop-local side effects.

## Implementation

- Shared utility layer: `include/ir.h`, `src/ir.cpp`.
- Semantic layer namespace: `sparseir::semantic`.
- Scheduled layer namespace: `sparseir::scheduled`.
- Lowering entrypoints:
  `lowerToSemanticProgram(...)`
  `lowerComputationToSemantic(...)`
  `scheduleProgram(...)`
  `scheduleCompute(...)`
  `lowerFirstComputationToScheduled(...)`.
- The active pipeline does not use the older `ir::Operation` model as the main program representation.

## Layer Responsibilities

- Shared `ir`:
  tensor metadata
  expression analysis metadata
  optimization metadata
  structured statement/expression nodes used by scheduled loop bodies.
- Semantic IR:
  compute meaning
  iterator classification
  sparse/dense source analysis
  output strategy derivation.
- Scheduled IR:
  executable loop tree
  block wrappers
  merge headers
  prologue/epilogue statements
  optimization annotations.

## Why the Split Matters

- Sparse legality checks depend on scheduled loop structure, not only source notation.
- Output behavior such as sparse union, intersection, sampled output, and dynamic row accumulation is easier to derive in semantic IR and easier to emit in scheduled IR.
- This division keeps codegen closer to rendering than to analysis.

## Primary Code Paths

- `include/ir.h`
- `include/semantic_ir.h`
- `src/semantic_ir.cpp`
- `src/ir.cpp`
