# 3.4.2 Code Blocking (Loop Tiling)

## Design

- Blocking is implemented as strip-mining of dense loops.
- The transformation is structural: it inserts a wrapper loop rather than annotating a loop and asking codegen to reinterpret it.
- This keeps the scheduled IR explicit and makes blocked schedules visible to later passes and tests.
- The system supports:
  1D blocking
  2D blocking for selected kernels.

## Implementation

- The key helper is `blockLoopByIndex(...)`.
- It finds a target dense loop by index name, creates a new `LoopKind::Block` wrapper, and moves the original dense loop underneath it.
- `configureBlockLoopEmission(...)` fills:
  block variable name
  block size
  trip count expression
  start and end variable names
  inner loop bound expressions.
- Codegen renders block loops through `LoopHeaderKind::Block`.

## Target Selection

- For dense-output, one-sparse one-dense cases such as SpMM, the default tiled index is usually the second output index.
- For kernels with one sparse and two dense inputs, `chooseBlockingIndex(...)` may prefer a dense non-output index when present.
- With `enable2DBlocking`, the pass can tile both:
  output row and output column loops
  or output row plus the chosen dense/reduction dimension, depending on kernel shape.

## Emitted Structure

- Wrapper loop index name is formed as `<index>_block`.
- Runtime upper bound is preserved on the wrapper.
- Codegen emits:
  block trip loop
  `<index>_start`
  `<index>_end`
  bounded inner loop over the original index.

## Limits

- Sparse loops are not directly blocked.
- Blocking is guarded by `blockingApplied` to avoid duplicate application.
- The current system is cache-oriented and heuristic, not analytically tuned in the compiler.
