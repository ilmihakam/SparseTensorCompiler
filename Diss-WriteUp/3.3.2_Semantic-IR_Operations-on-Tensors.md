# 3.3.2 Semantic IR: Operations on Tensors

## Design

- Semantic IR is the first compiler-owned representation that makes tensor operations explicit.
- It preserves algebraic meaning while adding sparse-compiler analysis results.
- The key design goal is to capture enough information to schedule the computation without committing to a loop nest too early.

## Implementation

- Semantic expression nodes:
  `TensorRead`
  `Constant`
  `ScalarRef`
  `BinaryExpr`
  `CallExpr`.
- Semantic statement nodes:
  `semantic::Declaration`
  `semantic::Call`
  `semantic::Compute`
  `semantic::Region`.
- `semantic::Compute` stores:
  `lhs`
  output tensor metadata
  input tensor metadata
  semantic RHS expression
  iterator graph
  free/reduction indices
  `ExpressionInfo`
  `OutputStrategy`.

## Algebraic Representation

- For dissertation purposes, the semantic IR can be written in a compact algebraic form that mirrors the actual node types without exposing C++ details.
- This is documentation notation, not a concrete parser or serialized file format.

```text
Program ::= Stmt*

Stmt ::= Decl(T : F<d0, ..., dn>, [i0, ..., in-1])
       | Compute(Out[free_indices] = Expr)
       | Call(f(arg0, ..., argm))
       | Region([T0, ..., Tk], [idx0, ..., idxp]) { Stmt* }

Expr ::= Read(T[idx0, ..., idxp])
       | Const(c)
       | Scalar(x)
       | Expr + Expr
       | Expr * Expr
       | Apply(f, Expr*)

F ::= Dense | CSR | CSC
```

- A declaration can therefore be written as:
```text
A : CSR<M, K>, [i, k]
B : Dense<K, N>, [k, j]
C : Dense<M, N>, [i, j]
```

- A semantic region with a compute and call can be written as:
```text
Region([A, B, C, D], [i, j]) {
  Compute(A[i, j] = B[i, j] * C[i, j] + D[i])
  Call(checkpoint(A))
}
```

- A standard SpMM semantic compute can be written as:
```text
Compute(C[i, j] = A[i, k] * B[k, j])
```
  with:
  free indices = `[i, j]`
  reduction indices = `[k]`
  inputs = `[A, B]`
  output = `C`.

- This notation maps onto the implementation as follows:
  `Decl(...)` -> `semantic::Declaration`
  `Compute(...)` -> `semantic::Compute`
  `Call(...)` -> `semantic::Call`
  `Region(...)` -> `semantic::Region`
  `Read/Const/Scalar/Apply` -> subclasses of `sparseir::Expr`.

## Operation Analysis

- Input tensors are collected from RHS tensor accesses in first-seen order.
- `analyzeExpression(...)` computes:
  root operator kind
  whether a function call appears
  fusion function name
  number of tensor accesses
  sparse input count
  dense input count
  sparse/dense input name lists.
- `deriveOutputStrategy(...)` classifies the result as:
  `DenseArray`
  `SparseFixedPattern`
  `HashPerRow`.

  <TensorName>: <Type>, [i, j, k]  (type in in CSC, CSR)
  for [<TensorNameList>] [<Iterator Index>] {
  compute A[i][j] = B[i][j] * C[i][j] + D[i]
  call <Function>(<TensorName>)
  }

## Output Strategy Cases

- Dense output:
  standard SpMV and dense-output SpMM.
- Sparse fixed pattern:
  pointwise sparse union/intersection
  sampled dense contraction where output reuses a sparse mask pattern.
- Hash-per-row:
  dynamic sparse accumulation, used for sparse-output contractions such as the implemented SpGEMM-style path.

## Why This Layer Matters

- Scheduling decisions depend on semantic facts such as which index is a reduction and whether multiple sparse sources imply union or intersection.
- These facts are easier to derive here than from the eventual loop tree.
