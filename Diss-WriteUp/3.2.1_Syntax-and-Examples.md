# 3.2.1 Syntax and Examples

## Design

- The surface syntax is intentionally compact and close to textbook index notation.
- Statement termination is explicit with `;` except for `for` regions, which contain their own body.
- Tensor format appears in declarations, not in compute statements.
- Operator precedence is minimal:
  `*` binds more tightly than `+`
  both are left-associative.

## Implementation

- Keywords in `src/lexer.l`:
  `compute`, `call`, `tensor`, `for`, `Dense`, `CSR`, `COO`, `CSC`, `ELLPACK`, `DIA`.
- Delimiters and operators are tokenized explicitly:
  `[ ] ( ) { } , ; : < > + * =`.
- `TOKEN_NUMBER` accepts integers and decimals via `[0-9]+(\.[0-9]+)?`.
- `TOKEN_IDENTIFIER` accepts `[a-zA-Z_][a-zA-Z0-9_]*`.
- Parser precedence in `src/parser.y`:
  `%left TOKEN_PLUS`
  `%left TOKEN_MULT`.

## Formal Grammar

- The current parser is implemented in Bison, but the equivalent EBNF below is a clearer dissertation form for the accepted surface syntax.
- This grammar is intentionally close to `src/parser.y` rather than a generalized mathematical language spec.

```ebnf
program         ::= statement_list
                  | ε

statement_list  ::= statement
                  | statement_list statement

statement       ::= declaration ";"
                  | computation ";"
                  | call_statement ";"
                  | for_statement

declaration     ::= "tensor" IDENTIFIER ":" tensor_type
                  | "tensor" IDENTIFIER ":" tensor_type "<" shape_list ">"

tensor_type     ::= "Dense"
                  | "CSR"
                  | "COO"
                  | "CSC"
                  | "ELLPACK"
                  | "DIA"

shape_list      ::= NUMBER
                  | shape_list "," NUMBER

computation     ::= "compute" tensor_access "=" expression

call_statement  ::= "call" IDENTIFIER "(" ")"
                  | "call" IDENTIFIER "(" argument_list ")"

for_statement   ::= "for" "[" identifier_list "]"
                        "[" index_list "]"
                        "{" "}"
                  | "for" "[" identifier_list "]"
                        "[" index_list "]"
                        "{" statement_list "}"

expression      ::= tensor_access
                  | function_call
                  | NUMBER
                  | IDENTIFIER
                  | expression "+" expression
                  | expression "*" expression

function_call   ::= IDENTIFIER "(" ")"
                  | IDENTIFIER "(" argument_list ")"

argument_list   ::= expression
                  | argument_list "," expression

tensor_access   ::= IDENTIFIER "[" index_list "]"

index_list      ::= IDENTIFIER
                  | index_list "," IDENTIFIER

identifier_list ::= IDENTIFIER
                  | identifier_list "," IDENTIFIER
```

## Core Forms

- Tensor declaration without shape:
```tc
tensor A : CSR;
```

- Tensor declaration with shape:
```tc
tensor B : Dense<80, 50>;
```

- Compute statement:
```tc
compute y[i] = A[i, j] * x[j];
```

- Sparse-dense matrix multiplication:
```tc
tensor C : Dense<100, 50>;
tensor A : CSR<100, 80>;
tensor B : Dense<80, 50>;
compute C[i, j] = A[i, k] * B[k, j];
```

- Call statement:
```tc
call checkpoint();
call optimize(A);
```

- Region form:
```tc
for [A, B] [i, j] {
    compute C[i, j] = A[i, j] + B[i, j];
}
```

## Notes from the Current Parser

- Parenthesized arithmetic is not modeled as a separate grammar production in the current `parser.y`.
- Function calls are available as expressions and as `call` statements.
- Empty `for` bodies are accepted.
- The standard single-compute path ignores non-compute constructs; the scheduled-program path preserves them.
