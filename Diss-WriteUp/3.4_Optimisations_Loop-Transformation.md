# 3.4 Optimisations: Loop Transformation

## Design

- Loop transformations operate on scheduled IR, not on source syntax and not directly on emitted C.
- This keeps legality checks close to the explicit loop structure and sparse dependencies.
- The optimization system mixes:
  correctness-preserving format alignment
  performance-oriented loop interchange
  performance-oriented blocking/tiling.
- Schedule order matters, so the system exposes explicit optimization ordering.

## Implementation

- Configuration lives in `opt::OptConfig`.
- Passes are implemented in `src/scheduled_optimizations.cpp`.
- Public entrypoints:
  `applyReordering(...)`
  `applyBlocking(...)`
  `applyLoopInterchange(...)`
  `applyOptimizations(...)` for both `scheduled::Compute` and `scheduled::Program`.
- The CLI in `src/main.cpp` maps flags such as:
  `--opt-interchange`
  `--opt-block=SIZE`
  `--opt-block-2d=SIZE` or `SIZExSIZE`
  `--opt-all=SIZE`

## Pass Ordering

- The passes mutate the scheduled loop tree in place and record results in `compute.optimizations`.

## Scope of the Current System

- Interchange and blocking are implemented over scheduled loop nests, not over arbitrary polyhedral dependence graphs.
- Program-level optimization is per statement; there is no cross-statement scheduling.
