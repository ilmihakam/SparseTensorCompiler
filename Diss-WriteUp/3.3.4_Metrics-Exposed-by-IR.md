# 3.3.4 Metrics Exposed by IR

## Design

- The IR carries analysis metadata that is useful for both optimization selection and code generation.
- These metrics are lightweight summaries, not a separate cost model.
- The repository uses them mainly to classify expressions, choose sparse-output paths, and annotate transformed schedules.

## Implementation

- `ir::ExpressionInfo` exposes:
  `rootOp`
  `isFused`
  `fusionFunction`
  `numTensorAccesses`
  `numSparseInputs`
  `numDenseInputs`
  sparse input names
  dense input names.
- `ir::LoopOptimizations` records:
  whether format-correctness reordering ran
  original/new order for reordering
  whether interchange ran
  whether blocking ran
  block size
  tiled index
  2D blocking flags and sizes.
- Scheduled compute metadata also exposes:
  `outputStrategy`
  `outputPattern`
  `patternSources`
  symbolic runtime bounds on loops.

## How the Metrics Are Used

- Semantic lowering uses tensor counts and root operator to derive output strategy and output pattern.
- Optimization passes update `LoopOptimizations` so the backend can emit comments and preserve schedule identity.
- `CodeGenerator` builds an `EmissionContext` from scheduled IR:
  sparse vs dense input inventory
  output behavior
  index bounds
  explicit runtime parameters
  whether nested sparse traversal exists.
- Inspection helpers in `src/ir.cpp` and `src/semantic_ir.cpp` render statements and scheduled loops for tests and debugging.

## What Is Not Present

- There is no global autotuning score or profitability model in the IR.
- Benchmark-time matrix feature extraction exists, but it is emitted into generated C and computed at runtime, not stored as IR fields.

## Primary Code Paths

- `include/ir.h`
- `src/ir.cpp`
- `src/semantic_ir.cpp`
- `src/codegen.cpp`
