# 3.3.3 Scheduled IR: Loop Tree over Sparse Formats

## Design

- Scheduled IR is the executable representation of the compiler.
- Each compute is lowered to a loop tree whose nodes already know how they should be emitted.
- The design avoids implicit codegen heuristics by storing explicit loop header metadata on the scheduled node itself.

## Implementation

- `scheduled::Compute` stores:
  output and input tensor metadata
  semantic RHS expression
  `ExpressionInfo`
  `OutputStrategy`
  `OutputPatternKind`
  sparse-pattern source names
  prologue and epilogue statements
  optimization metadata
  `rootLoop`
  `fullyLowered`.
- `scheduled::Loop` stores:
  logical index name
  static and symbolic bounds
  `LoopKind`
  `LoopHeaderKind`
  binding variables/expressions
  sparse iterator metadata
  merge metadata
  block metadata
  children
  pre/post IR statements
  `isExternallyBound`.

## Loop Header Kinds

- `DenseFor`:
  emits a standard counted `for` loop.
- `SparseIterator`:
  emits pointer-driven traversal over CSR or CSC structure.
- `SparseMerge`:
  emits merge-based traversal for union/intersection over multiple sparse inputs.
- `Block`:
  emits strip-mined wrappers with start/end variables and bounded inner loops.

## Lowering Details

- `deriveLoopLoweringSpec(...)` decides, per iterator:
  dense vs sparse
  merge strategy
  driver tensor
  parent index
  accumulation strategy.
- `buildGenericLoopNest(...)` recursively builds the loop tree.
- Special accumulation paths are inserted structurally:
  scalar accumulator for sampled dense contraction
  workspace/hash-style accumulator for dynamic sparse output.
- `markExternallyBound(...)` marks loops already provided by an enclosing `Region`, so codegen can omit duplicate loop headers.

## Consequence

- By the time codegen runs, loop emission is mostly a rendering problem:
  it reads scheduled header metadata instead of reconstructing traversal semantics from expressions.
