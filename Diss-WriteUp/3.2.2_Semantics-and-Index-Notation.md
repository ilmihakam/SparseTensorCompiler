# 3.2.2 Semantics and Index Notation

## Design

- The language uses index notation with implicit summation.
- Indices on the left-hand side define the output iteration space.
- Indices that appear only on the right-hand side define reduction dimensions.
- Sparse semantics are storage-aware:
  in CSR, the second logical index is sparse
  in CSC, the first logical index is sparse.
- Output behavior is derived from the combination of input sparsity, root operator, and whether the output tensor itself is sparse.

## Implementation

- Semantic expression lowering converts AST nodes to:
  `TensorRead`
  `Constant`
  `ScalarRef`
  `BinaryExpr`
  `CallExpr`.
- `lowerComputationToSemantic(...)` computes:
  output tensor use
  input tensor set
  free indices
  reduction indices
  iterator graph
  `ExpressionInfo`
  `OutputStrategy`.
- Reduction inference is syntactic:
  collect all RHS indices
  remove indices already on the LHS
  preserve first-seen order.
- Sparse axis detection is implemented by `isSparseAccessPosition(...)`:
  CSR is sparse at access position `1`
  CSC is sparse at access position `0`.

## Semantic Consequences

- `compute C[i, j] = A[i, k] * B[k, j];`
  free indices: `i`, `j`
  reduction index: `k`.
- `compute C[i, j] = A[i, j] + B[i, j];`
  no reduction index
  if `A`, `B`, and `C` are sparse, this becomes fixed-pattern sparse union behavior.
- `compute C[i, j] = A[i, j] * B[i, j];`
  with sparse inputs and sparse output, this becomes fixed-pattern sparse intersection behavior.

## Important Current Semantics

- Static dimensions are optional; missing dimensions fall back to runtime-bound inference where possible.
- Scalar identifiers on the RHS are allowed and lowered as `ScalarRef`.
- Function-call semantics are carried through the expression tree, but only selected functions are inlined specially during codegen.
- Non-`CSR`/`CSC` formats parsed by the frontend do not currently carry distinct sparse semantics into active lowering.
