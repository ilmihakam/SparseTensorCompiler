# 3.3.1 Tensor and Format Model (CSR/CSC)

## Design

- Tensors are represented by metadata rather than concrete storage arrays during IR construction.
- The crucial design choice is to encode storage format and source-level index order together.
- This lets the compiler answer:
  which logical axis is sparse
  which runtime dimension should bound a loop
  which pointer/index arrays a sparse iterator should use later.

## Implementation

- `ir::Tensor` stores:
  `name`
  `format`
  `dims`
  `indices`.
- `ir::Format` currently has three active variants:
  `Dense`
  `CSR`
  `CSC`.
- Source declarations are collected by `collectDeclarations(...)`.
- Format strings are normalized by `stringToFormat(...)`.
- Shape strings are parsed to integer dimensions by `parseShape(...)`.

## CSR and CSC Interpretation

- CSR:
  row dimension is the outer dense axis
  column dimension is the inner sparse axis
  runtime arrays later used are `row_ptr`, `col_idx`, `vals`.
- CSC:
  column dimension is the outer dense axis
  row dimension is the inner sparse axis
  runtime arrays later used are `col_ptr`, `row_idx`, `vals`.
- This interpretation drives `isSparseAccessPosition(...)` during semantic analysis and sparse loop configuration during scheduling.

## Bound Inference

- Static dimensions are preferred when declared.
- Sparse dimensions can also be derived from loaded matrix metadata:
  row loops map to `tensor->rows`
  column loops map to `tensor->cols`.
- Dense fallback parameters use symbolic names such as `N_i`, `N_j`, `K`.
