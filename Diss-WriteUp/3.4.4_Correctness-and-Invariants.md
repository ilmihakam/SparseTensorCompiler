# 3.4.4 Correctness and Invariants

## Design

- The optimization system is built around explicit invariants on scheduled loop structure.
- The repository does not prove transformations formally, but it does encode legality guards and regression tests around the main sparse cases.

## Implementation Invariants

- Dense-loop blocking invariant:
  only `LoopKind::Dense` loops may be blocked.
- Sparse dependency invariant for interchange:
  the pass refuses sparse-on-sparse interchange patterns that would break data dependence.
- Block-loop identity invariant:
  synthetic block loops are recognized by names containing `"_block"`.
- Runtime-bound invariant:
  when a loop is wrapped by a block loop, the wrapper retains the original symbolic upper bound.
- Metadata invariant:
  reordering/blocking flags prevent repeated application of the same transformation instance.

## Legality Checks in the Current Code

- `canSwapLoops(...)` prevents root-child swaps when sparse structure would be invalid.
- `isInterchangeLegal(...)` rejects the case where both the middle and inner loops are sparse.
- `blockLoopByIndex(...)` exits if the chosen target is not a dense loop.
- `findLoopSlotByName(...)` ensures transformations target an explicit loop identity in the tree rather than relying on position alone.

## Behavioral Invariants After Transformation

- Loop bodies remain attached to a valid scheduled subtree.
- Optimization order is explicit and reproducible.
- Program-level optimization preserves region nesting by applying transformations recursively inside each region body.
- Generated kernels still carry the same input/output signatures for a given scheduled compute, even when loop structure changes.

## Validation Strategy

- Unit tests check schedule shape and metadata.
- Codegen tests check that transformed schedules still emit balanced and expected C.
- Correctness tests compare transformed kernels against baselines/reference implementations.
