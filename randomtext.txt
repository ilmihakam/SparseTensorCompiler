# Honours Project Overview
Machine learning matrices are filled with near-zero or zero values, which has prompted exploration into sparse matrices that avoid computing on these values. However, unlike their dense equivalents, sparse tensors struggle to take advantage of modern hardware, resulting in massive inefficiencies.

Compiler infrastructure struggles to fill this gap, as each different format requires a different set of optimization techniques.  In this project, you will look at optimization strategies for common operations on these data structures.

In this project, you will pick an existing, underexplored sparse format and optimize it. The key outcomes will be the following:
  - Build simple matrix operations (add, multiply) in your sparse format
  - Optimise those matrix operations by hand
  - Build a simple compiler infrastructure that generates these optimised matrix
     operations (students looking for an easy project should end here)
  - Build a simple optimiser for sequences of these operations to enable operator fusion on your datastructure of choice (see suggestions below).

Data structure choice is left up to you, but the following suggestions would make good projects: COO (coordinate list), CSR (compressed sparse row), CSC (compressed sparse column), ELL (ELLPACK), and DIA (Diagonal format)  --- more suggestions can be found in section 2.1.3 of https://dl.acm.org/doi/pdf/10.1145/3571157 and I'm open to other suggestions for formats.

Primary goal of the project: Knowing which optimization techniques are better for specific formats (i.e. COO, CSR, CSC and ELL)

# Developing my own DSL
I need to develop my Domain-Specific Language for Sparse Compiler which needs to have:
Operator: + (addition), some function f (to be expressed as a Machine Learning model), call method, index notation (implicit, explicit), * (multiplication). Making benchmarking easy, DSL

The DSL uses explicit type-enforced tensor declarations (Dense, CSR, COO, CSC, ELLPACK, DIA)

# Lowering
The DSL needs to be lowered to an AST in which the AST is lowered further into the IR where the IR will be tested for code generation, I'm looking to target with C++/LLVM with the IR.

# Mapping (Type Checking)
Semantic Analysis occurs to check the properties of the tensors and map them based on the properties to a TensorType.

# Sparse Tensor Base Format

The Sparse IR needs to have:

<TensorName>: <Type>, [i, j, k]  (type in in CSC, CSR)
for [<TensorNameList>] [<Iterator Index>] {
compute A[i][j] = B[i][j] * C[i][j] + D[i]
call <Function>(<TensorName>)
}

## Phase 2: Extended Lexer MVP
**Additional tokens needed:**
- **Keywords**: `call`, `tensor`, `for`
- **Delimiters**: `{`, `}`, `:`
- **All existing tokens**: `compute`, `+`, `*`, `=`, `[`, `]`, `(`, `)`, `,`, `;`

**Tests**: Verify all tokens including new ones for IR compatibility

## Phase 3: Extended Parser MVP
**New grammar rules to support IR lowering:**
```ebnf
program         ::= statement*
statement       ::= computation ';' | call_statement ';' | declaration ';' | for_statement
computation     ::= 'compute' tensor_access '=' expression
call_statement  ::= 'call' IDENTIFIER '(' argument_list? ')'
declaration     ::= 'tensor' IDENTIFIER ':' tensor_type ('<' shape_list '>')?
tensor_type     ::= 'Dense' | 'CSR' | 'CSC' 
shape_list      ::= NUMBER (',' NUMBER)*
for_statement   ::= 'for' '[' identifier_list ']' '[' index_list ']' '{' statement_list '}'
```

**Tests**: Each new construct + combinations that match IR structure

## Phase 4: IR Lowering Validation
**Test examples that can lower to IR:**
```cpp
// DSL Input:
tensor A : CSR<100, 50>;
tensor B : Dense<50, 20>;
tensor C : Dense<100, 20>;
for [A, B, C] [i, j, k] {
    compute C[i, j] = A[i, k] * B[k, j];
    call optimize(C);
}

// Should map to IR structure exactly
```
